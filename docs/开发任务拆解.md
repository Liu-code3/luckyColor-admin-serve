# LuckyColor Admin Serve 开发任务拆解

## 1. 目标范围

本清单面向当前 `NestJS + Prisma + MySQL` 后端仓库，目标是将项目逐步建设为可快速集成的多租户 SaaS 后台服务，覆盖以下能力：

- 系统功能：用户管理、角色管理、菜单管理、部门管理、字典管理、系统配置、通知公告。
- 权限管理：动态路由、按钮权限、角色权限、数据权限。
- 基础设施：国际化、多布局、暗黑模式、全屏、水印、接口文档、代码生成器。
- 多租户：支持租户模式与租户隔离。

## 2. 设计原则

- 低耦合：按领域拆分模块，模块之间通过明确的服务接口、DTO、事件或共享能力协作，避免跨模块直接读写内部实现。
- 高内聚：每个模块只负责一个清晰领域，例如 `users` 只处理用户领域，`roles` 只处理角色领域。
- 可扩展：优先抽象公共基类、查询模型、权限判定模型、租户上下文与配置中心，避免后续新增模块时重复造轮子。
- 易维护：控制器只负责输入输出，业务逻辑进入服务层，数据库访问集中在仓储/Prisma 访问层，跨领域通用能力沉淀到 `shared`。
- 多租户优先：凡是业务数据表、查询入口、缓存键、日志与审计能力，默认都要考虑 `tenantId`。

## 3. 推荐结构

建议逐步演进为如下结构：

```text
src/
  app.module.ts
  main.ts
  modules/
    system/
      users/
      roles/
      menus/
      departments/
      dictionaries/
      system-configs/
      notices/
    iam/
      auth/
      permissions/
      data-scopes/
    tenant/
      tenants/
      tenant-packages/
      tenant-context/
    platform/
      i18n/
      preferences/
      watermark/
      docs/
      codegen/
  shared/
    api/
    auth/
    tenant/
    database/
    pagination/
    decorators/
    guards/
    interceptors/
    filters/
    utils/
```

说明：

- 当前仓库已有 `auth`、`users`、`menus`、`dictionary` 基础模块，可按上面结构逐步迁移，不要求一次性重构完。
- UI 相关能力如多布局、暗黑模式、全屏、水印，在本后端仓库中主要负责配置、偏好保存、字典下发、权限控制与接口支持。

## 5. 推荐实施顺序

建议按以下顺序推进，避免返工：

1. `P0` 架构基线与公共能力。
2. `P1` 多租户内核。
3. `P2` 认证与权限内核。
4. `P3` 系统管理模块。
5. `P4` 基础设施能力。
6. `P5` 测试、索引、发布与回归。

## 6. 执行要求

- 每完成一个小任务，必须按 [Gitflow 提交规范](./Gitflow提交规范.md) 新建分支并提交一次。
- 每次提交只解决一个清晰目标，不把多个无关任务混在同一个提交里。
- 每次推进本清单中的任务时，任务完成后必须立即提交一次，不允许累积多个任务后再统一提交。
- 每次修复开发过程中发现的问题、回归或阻塞项时，也必须单独提交一次。
- 涉及 Prisma 模型变更时，同步维护 schema、生成产物、种子数据与相关查询逻辑。
- 涉及权限、租户、数据范围的任务，必须补至少一个越权或隔离场景验证。

## 7. P0 到 P2 实施顺序

### 阶段 A. 先打基础骨架

目标：先统一项目结构、配置入口、公共返回与查询模型，避免后面做租户和权限时四处返工。

建议顺序：

1. `P0-01` 梳理模块边界与目录结构。
2. `P0-02` 统一返回体、异常处理、错误码。
3. `P0-03` 统一 DTO、分页、排序白名单。
4. `P0-04` 建立统一配置模块与环境变量校验。
5. `P0-07` 建立基础枚举与常量中心。
6. `P0-08` 建立提交前门禁，固定 `pnpm typecheck` 与 `pnpm build`。
7. `P0-09` 预留测试目录与最小样例。

原因：

- `P1` 和 `P2` 都依赖统一的配置、异常、分页、枚举与目录规范。
- 如果先做租户和权限，再回头改公共层，改动面会明显扩大。

### 阶段 B. 再打数据库与公共字段基线

目标：先把 Prisma 与通用字段标准化，让后续模型扩展有统一落点。

建议顺序：

1. `P0-05` 统一 Prisma 访问封装。
2. `P0-06` 约定业务表通用字段。
3. `P0-10` 梳理种子数据机制。

原因：

- 租户、角色、菜单、部门、配置这些模型都要复用通用字段。
- 如果没有先统一 Prisma 访问层，后面容易在每个模块里重复写租户过滤、审计字段和软删除逻辑。

### 阶段 C. 优先完成多租户内核

目标：先把“谁属于哪个租户、请求属于哪个租户、数据如何按租户隔离”做对，再做权限。

建议顺序：

1. `P1-01` 设计 `Tenant` 表。
2. `P1-02` 设计 `TenantPackage` 或套餐模型。
3. `P1-03` 明确租户识别方式。
4. `P1-04` 实现租户上下文注入。
5. `P1-05` 实现统一租户过滤中间层。
6. `P1-06` 明确平台管理员与租户管理员边界。
7. `P1-07` 实现租户初始化流程。
8. `P1-08` 实现租户状态统一拦截。
9. `P1-09` 补充租户审计日志。
10. `P1-10` 完成租户隔离回归清单。

原因：

- 权限体系中的用户、角色、菜单、数据权限都要挂在租户边界之内。
- 如果先做 `P2`，后续再补租户，会导致角色关系、登录态、菜单输出、数据权限全部重做。

### 阶段 D. 先完成认证闭环

目标：让系统先具备“能登录、能识别是谁、能识别属于哪个租户”的基础身份能力。

建议顺序：

1. `P2-01` 重构登录流程。
2. `P2-02` 接入密码加密与密码更新策略。
3. `P2-12` 增加登录、登出与安全审计记录。

原因：

- 登录是权限体系入口。
- 密码策略和审计应尽早接入，避免继续基于不安全模型扩展。

### 阶段 E. 再完成角色与权限点模型

目标：先把“授权对象”和“权限资源”模型立住，再做接口守卫和前端输出。

建议顺序：

1. `P2-03` 设计角色及关联关系模型。
2. `P2-04` 统一菜单权限、按钮权限、接口权限的权限点编码。
3. `P2-05` 建立权限装饰器与守卫。
4. `P2-11` 统一无权限、租户禁用、角色失效等异常返回。

原因：

- 角色关系和权限点编码是整个授权体系核心。
- 守卫与异常返回依赖前面的模型定义，否则后续会反复改判定规则。

### 阶段 F. 最后完成前端可消费的权限输出与数据权限

目标：把权限能力真正变成前端和业务查询可消费的接口输出。

建议顺序：

1. `P2-06` 输出登录态与权限快照。
2. `P2-07` 定义动态路由返回结构。
3. `P2-08` 增加按钮权限查询接口。
4. `P2-09` 设计数据权限模型。
5. `P2-10` 实现数据权限拼接能力。

原因：

- 动态路由和按钮权限依赖角色与权限点模型稳定。

## 8. 面向未来兼容 Java / Go 服务的约束

### 8.1 结论

当前项目继续使用 `NestJS` 作为主服务没有问题，但从现在开始就要按“未来可能增加 `Java` 或 `Go` 独立服务”的方式约束架构。

目标不是现在就拆微服务，而是先把边界、协议、租户、权限、数据模型收敛好。这样未来新增其他语言服务时，代价是“新增一个服务并接入体系”，而不是“重写整套系统”。

### 8.2 现在必须坚持的架构规则

- 所有模块必须有清晰边界，禁止跨模块直接操作对方内部实现。
- 控制器只负责协议转换，业务逻辑进入服务层，数据库访问集中在仓储或 Prisma 访问层。
- 租户识别、权限校验、数据权限拼接必须沉淀为公共能力，禁止散落在各业务模块里手写。
- 前后端接口结构保持稳定，接口字段、错误码、分页结构尽量统一。
- 所有核心业务表统一通用字段规范，至少包含 `tenantId`、状态、审计字段。
- 模块之间尽量通过服务接口、应用服务、事件机制协作，避免直接跨表查询。
- 配置、字典、权限点、租户状态等基础能力要有统一出口，不要每个模块自定义一套。
- 新增能力优先放入已有模块边界内，避免出现“为了快先塞进去，后面再整理”的临时实现。

### 8.3 为跨语言演进预留的实现要求

- 尽早接入 `OpenAPI / Swagger`，让未来 `Java` 或 `Go` 服务更容易对齐接口契约。
- 统一认证与鉴权协议，例如 Token 结构、用户上下文、租户上下文、权限快照格式。
- 统一审计日志格式，保证不同语言服务接入后日志字段仍可聚合检索。
- 统一分页、排序、筛选协议，避免未来多个后端服务返回风格不一致。
- 统一错误码和异常响应结构，避免前端需要按服务类型分别兼容。
- 对租户、角色、菜单、部门、字典等主数据建立明确主权归属，避免多个服务各管一份。
- 对后续可能拆分的能力优先考虑事件驱动或异步通知方式，减少强同步调用耦合。
- 重要基础能力要先文档化，例如权限点命名规范、租户解析规则、数据权限范围定义。

### 8.4 未来适合优先拆出去练手的服务

这些能力边界相对清晰，适合以后用 `Java` 或 `Go` 单独实现：

- 文件上传、文件转存、对象存储管理。
- 站内信、短信、邮件、消息推送。
- 定时任务、批处理、报表导出。
- 操作日志、登录日志、审计归档。
- 代码生成器、模板渲染、脚手架输出。
- 搜索、全文检索、统计计算类服务。

原因：

- 这些模块通常和“登录主链路”“权限主链路”“租户核心模型”耦合较弱。
- 拆出去后更容易通过 HTTP、消息队列或任务队列接入，不容易破坏主系统。

### 8.5 不建议优先拆分的核心模块

这些模块在项目前期不建议优先拆到 `Java` 或 `Go`：

- 用户管理。
- 角色管理。
- 菜单管理。
- 权限中心。
- 租户中心。
- 数据权限中心。

原因：

- 这些模块是整个后台系统的权限与租户地基。
- 一旦过早拆分，会显著增加认证同步、权限一致性、事务边界、接口联调和排障成本。
- 当前阶段更适合先在一个主服务内把领域模型打稳，再考虑局部演进。

### 8.6 当前阶段的实际策略

建议策略如下：

1. 当前阶段继续以 `NestJS` 作为单体主服务推进。
2. 优先把 `P0`、`P1`、`P2` 做规范，尤其是租户上下文、权限模型、统一协议。
3. 在 `P4` 基础设施能力建设时，优先挑选一个低耦合模块作为未来跨语言拆分试点。
4. 等接口契约、日志规范、错误码规范稳定后，再考虑引入 `Java` 或 `Go` 独立服务。

### 8.7 开发过程中的自检问题

每做一个模块前，先问自己以下问题：

- 这个模块是否直接依赖了别的模块内部表结构？
- 这个模块的租户过滤是否通过公共能力完成？
- 这个模块的权限校验是否通过统一守卫或权限点完成？
- 这个模块未来如果迁到 `Java` 或 `Go`，是否还能保持接口契约不变？
- 这个模块是否把返回结构、错误码、分页协议做成了项目统一格式？

如果以上问题有两项以上回答为“否”，说明当前实现方式会提高未来跨语言演进成本，需要先调整设计再继续开发。

## 9. 开发红线清单

本节是执行级约束。后续开发时，默认以“禁止”为主，除非有充分理由并且明确记录。

### 9.1 模块边界红线

- 禁止在 `users` 模块中直接读写 `roles`、`menus`、`tenant` 模块的内部实现细节。
- 禁止跨模块直接依赖对方的 Prisma 查询细节或表结构拼装逻辑。
- 禁止把多个领域的业务逻辑塞进同一个 service 中。
- 禁止为了赶进度，把临时逻辑长期放在 `controller` 中不下沉。
- 禁止在 `common` 或 `shared` 里堆放强业务语义代码，公共层只放真正通用能力。

### 9.2 租户隔离红线

- 禁止在业务查询中遗漏 `tenantId` 过滤。
- 禁止在控制器或业务模块里零散手写租户解析逻辑，统一走租户上下文能力。
- 禁止把平台管理员绕过逻辑散落在各模块中，必须统一收敛到权限或租户判定层。
- 禁止出现“某些接口有租户过滤、某些接口没有”的半成品状态进入主干。
- 禁止新增业务表时不评估是否需要 `tenantId` 字段。

### 9.3 权限控制红线

- 禁止直接依赖前端传来的角色、权限码、数据范围作为最终可信结果。
- 禁止在控制器里写死角色名、菜单名来判断权限。
- 禁止把按钮权限、菜单权限、接口权限定义成多套不兼容规范。
- 禁止未接入统一守卫的敏感接口直接上线，例如删除、重置密码、停用、授权。
- 禁止数据权限逻辑散落在列表查询中手工拼接且没有统一出口。

### 9.4 数据模型红线

- 禁止核心业务表缺少统一审计字段规范。
- 禁止字段命名风格混乱，例如同类语义同时存在 `createTime`、`createdAt`、`gmtCreate`。
- 禁止在没有索引评估的情况下随意增加高频筛选字段。
- 禁止在 DTO、数据库模型、接口返回体之间长期保持命名错位。
- 禁止把“字典类型”“字典项”“配置项”“权限点”这类主数据做成语义混杂的大杂烩表。

### 9.5 接口契约红线

- 禁止同类分页接口返回结构不一致。
- 禁止错误响应格式随模块变化。
- 禁止列表筛选参数同名不同义，或同义不同名。
- 禁止为了图省事直接把数据库实体原样透出给前端。
- 禁止未评估兼容性就随意修改已对前端开放的核心字段。

### 9.6 工程质量红线

- 禁止一个提交混入多个不相关任务。
- 禁止未经 `pnpm typecheck` 与 `pnpm build` 验证直接提交功能改动。
- 禁止 Prisma 模型改了但不更新生成产物、种子数据或关联查询。
- 禁止上线前没有验证越权、越租户、角色失效、租户停用等关键场景。
- 禁止把本地临时调试代码、临时日志、无用注释长期留在主干。

### 9.7 跨语言演进红线

- 禁止把认证协议、错误码、租户上下文写成只适配 `NestJS` 的私有风格。
- 禁止出现只有代码没有契约文档的核心接口。
- 禁止多个服务分别维护各自版本的用户、角色、租户真相数据。
- 禁止在没有边界梳理的情况下贸然把用户、角色、菜单、租户核心拆成独立服务。
- 禁止为了练习 `Java` 或 `Go` 而优先拆最核心、最耦合的权限主链路。

## 10. 模块协作边界

本节用于明确后续模块之间“能依赖什么、不能依赖什么”。

### 10.1 `tenant`

职责：

- 负责租户主数据、套餐、租户状态、租户初始化。
- 负责租户上下文解析与租户状态判定。

可以提供：

- 当前租户上下文。
- 租户状态校验结果。
- 租户初始化服务。
- 平台管理员与租户管理员边界判断。

禁止承担：

- 用户登录流程细节。
- 角色授权逻辑。
- 菜单构建逻辑。

### 10.2 `iam`

职责：

- 负责认证、权限点、角色授权、数据权限规则。
- 负责登录态、权限快照、接口守卫、统一鉴权。

可以依赖：

- `tenant` 提供的当前租户上下文与租户状态。
- `system` 提供的用户、角色、菜单、部门主数据查询服务。

禁止承担：

- 直接管理业务公告、系统配置、字典项内容。
- 直接承载租户套餐与租户生命周期逻辑。

### 10.3 `system`

职责：

- 负责用户、角色、菜单、部门、字典、系统配置、通知公告等系统域模型。
- 对外提供标准化的查询、写入、授权所需主数据接口。

可以依赖：

- `tenant` 的租户上下文与租户隔离能力。
- `iam` 的权限校验装饰器、守卫、数据权限能力。

禁止承担：

- 登录鉴权协议实现。
- Token 签发与认证主链路。
- 跨模块散落式权限判断。

### 10.4 `platform`

职责：

- 负责接口文档、国际化资源、用户偏好、水印配置、代码生成器等基础设施型能力。
- 负责服务于前端集成体验的通用支撑接口。

可以依赖：

- `system` 的字典、配置等主数据。
- `iam` 的权限能力。
- `tenant` 的租户上下文。

禁止承担：

- 核心认证逻辑。
- 用户、角色、租户主数据的最终真相管理。

### 10.5 `shared`

职责：

- 只承载跨模块复用的技术能力，例如异常、响应、分页、装饰器、守卫基类、工具函数。

禁止承担：

- 任何具体领域规则。
- 任何只属于单个业务模块的特化逻辑。

## 11. 每次开始编码前的检查顺序

为了避免偏航，开始一个小任务前先按下面顺序检查：

1. 这个任务属于哪个模块边界。
2. 这个任务是否涉及租户隔离。
3. 这个任务是否涉及权限点或数据权限。
4. 这个任务是否需要新增或调整表结构。
5. 这个任务是否会影响前端接口契约。
6. 这个任务是否需要补种子数据或字典数据。
7. 本次提交标题准备用什么中文描述。

如果第 2、3、4、5 任一项答案是“是”，必须在编码前先补齐设计点，不要直接开写。
