# Gitflow 提交规范

## 1. 说明

本规范用于约束本项目后续开发节奏：

- 后端项目每次新增功能或修改功能后，必须完成一次 Git 提交。
- 每完成一个小任务，就进行一次提交。
- 使用 `gitflow` 思路管理分支。
- `git commit message` 使用简洁中文，不写冗长说明。
- `docs/` 目录属于项目约束的一部分，任务拆解、流程规范与执行要求发生变化时必须进入 Git 版本控制。

## 2. 分支模型

- `main`：生产稳定分支，只接收发布或热修复合并结果。
- `develop`：日常集成分支，所有功能分支从这里切出并合回。
- `feature/<模块>-<任务>`：功能开发分支，对应一个小任务或一组强相关小任务。
- `release/<版本号>`：发版分支，用于提测、回归、修正文案与版本信息。
- `hotfix/<问题>`：线上紧急修复分支，从 `main` 切出，修复后同时回合 `main` 与 `develop`。

## 3. 小任务开发流程

1. 切换到 `develop` 并同步最新代码。
2. 从 `develop` 新建功能分支，例如：`feature/users-page-query`。
3. 只完成一个任务清单中的明确小项，例如 `P3-U02 用户分页查询`。
4. 本地自检至少执行：

```powershell
pnpm typecheck
pnpm build
```

5. 如果涉及 Prisma 变更，额外执行：

```powershell
pnpm prisma:generate
```

6. 自检通过后提交一次，提交说明使用简洁中文。
7. 推进 `docs/开发任务拆解.md` 中的任一任务时，该任务完成后必须立即提交一次。
8. 修复开发过程中发现的问题、回归或阻塞项时，也必须提交一次，不允许只改不提。
9. 功能分支完成后合并回 `develop`。
10. 到达一个可提测阶段时，从 `develop` 切 `release/<版本号>`。
11. 线上问题从 `main` 切 `hotfix/<问题>`，修复后同时回合 `main` 与 `develop`。

## 4. 提交要求

- 后端项目无论是新增功能还是修改已有功能，都不能跳过提交。
- 一个小任务对应一个提交，避免“顺手改一堆”。
- 推进 `docs/` 目录下的任务拆解文档时，每完成一个任务必须提交一次。
- 修复 Bug、回归、测试失败、构建失败等问题时，每完成一次修复也必须提交一次。
- 提交信息只写结果，不写过程，不堆砌技术细节。
- 提交标题建议控制在 8 到 20 个中文字符内。
- 提交前先确认没有把无关文件一并提交。
- 涉及数据库字段调整时，提交内容必须包含相关 schema 与代码适配。
- 涉及权限与租户逻辑时，提交前必须自行验证是否存在越权风险。

## 5. 中文提交格式

推荐格式：

```text
<模块>：<动作><结果>
```

可直接使用的示例：

```text
用户管理：完成分页查询
用户管理：补充重置密码
角色管理：增加数据权限
菜单管理：生成路由树
部门管理：支持树形查询
字典管理：拆分类型与字典项
系统配置：增加缓存读取
通知公告：支持定时发布
权限中心：接入按钮权限
权限中心：增加数据范围校验
租户中心：新增租户上下文
租户中心：隔离用户查询
基础设施：接入接口文档
代码生成：增加表结构读取
```

不建议使用的提交信息：

```text
fix
update
修改代码
继续开发
提交一下
各种优化
```

## 6. 分支命名示例

```text
feature/user-create-api
feature/role-data-scope
feature/menu-route-tree
feature/tenant-context
feature/swagger-bootstrap
release/v1.0.0
hotfix/login-token-check
```

## 7. 完成定义

一个小任务可以提交，至少要满足以下条件：

- 功能代码已完成。
- 相关 DTO、校验、权限或租户逻辑已同步补齐。
- 本地构建与类型检查通过。
- 没有明显的越权、空指针、字段遗漏问题。
- 提交说明可让人一眼看懂本次改动目的。

## 8. 每次开始开发前先看

- 本次要做的是哪个任务编号。
- 这个任务依赖哪些前置任务。
- 当前任务应该放在哪个模块目录下。
- 是否涉及租户隔离、权限校验、数据范围。
- 本次提交标题准备写什么中文内容。
